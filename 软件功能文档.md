# AI 短剧剧本生成系统 — 功能文档

> 基于 Gemini 2.5 Flash + PySide6 的多阶段 AI 剧本创作工具

---

## 第一部分：处理流程

### 总体流程概览

```
一句话梗概 (Sparkle)
    │
    ▼
┌─────────────────┐
│ Phase 1: 创世    │  苏格拉底盘问 → 世界观变量提炼
└────────┬────────┘
         ▼
┌─────────────────┐
│ Phase 2: 人物    │  AI 角色生成 → 关系网络构建
└────────┬────────┘
         ▼
┌─────────────────┐
│ Phase 3: 骨架    │  CPG 因果图生成 → 节点编辑/拆分/级联改写
└────────┬────────┘
         ▼
┌─────────────────┐
│ Phase 4: 血肉    │  多人格盲视变异 → Beat 选择确认
└────────┬────────┘
         ▼
┌─────────────────┐
│ Phase 5: 扩写    │  Beat → 标准剧本格式
└────────┬────────┘
         ▼
┌─────────────────┐
│ Phase 6: 锁定    │  审阅 → 导出
└─────────────────┘
```

### Phase 1：创世阶段

**目的**：从一句话梗概出发，通过追问锁定故事的「真理锚点」。

**流程**：

1. **输入 Sparkle**：用户输入一句话故事梗概（如「程序员意外进入自己写的游戏世界」）
2. **AI-Call-1 苏格拉底盘问**（温度 0.4）：
   - AI 扮演「世界观架构师」，针对 6 个维度生成 5-8 个追问
   - 维度覆盖：时空规则、核心冲突根源、角色动机链、关键道具/能力、终局标准、情感基调
   - 每个问题附带「为什么问这个」的理由
3. **用户逐一回答**：QA 面板支持未回答高亮提示、自动滚动定位
4. **AI-Call-2 世界观变量提炼**（温度 0.2）：
   - 将散碎问答整理为结构化「世界观变量表」
   - 变量分类：世界规则、角色设定、道具能力、社会制度、终局条件
   - 自动检测变量间矛盾并标记
   - 输出暂定标题和终局条件

**输出物**：世界观变量表（真理锚点）、终局条件、暂定标题

### Phase 2：人物阶段

**目的**：构建支撑故事冲突的角色阵容和关系网络。

**流程**：

1. **AI-Call-1.5 角色自动生成**（温度 0.5）：
   - 用户设定角色数量（如 5 个）
   - AI 根据 Sparkle + 世界观设计角色阵容
   - 角色分级：A级（核心）/ B级（重要）/ C级（工具）
   - A级角色额外设计：成长环境、压力反应、核心恐惧、核心渴望
   - 同时生成角色关系网络
2. **手动编辑**：编辑角色属性（姓名、性格、动机、外貌等）
3. **关系管理**：
   - 表格视图：直接编辑关系类型和描述
   - 力导向图视图：可视化拖拽交互，双向同步
4. **支持多轮追加**：可多次调用 AI 追加新角色，自动排除已有角色避免重复

**输出物**：角色列表（含分级）、角色关系网络

### Phase 3：骨架阶段

**目的**：生成 CPG（因果概率图）骨架——整个剧本的逻辑脊柱。

**流程**：

1. **参数配置**：设定总集数、每集时长、风格模式（短剧/传统影视）、题材预设
2. **AI-Call-3 CPG 骨架生成**（温度 0.6）：
   - ≤30 集：单次生成全部节点
   - \>30 集：两遍生成法（Pass 1 全局大纲 → Pass 2 逐阶段展开）
   - 每个节点对应一集，包含：标题、环境、角色、3-5 个事件摘要、情感基调、结尾钩子
   - 自动生成因果连线（直接因果/间接影响/情感驱动）
   - 严格遵循 Hauge 六阶段比例分配
3. **节点交互编辑**：
   - 双击节点打开详情对话框（支持手动编辑全部 6 个字段）
   - Chat 对话模式：与 AI 讨论节点内容，确认后执行修改
   - Quick-Regen：选择戏剧方向 + 结构调整，一键重生成
   - BVSR 重写：以指定人格视角完全重构
   - 节点拆分：将一集拆为 2-5 个片段，AI 补全/扩写
   - 级联改写：修改一章后自动调整后续 N 章
4. **版本管理**：每次修改自动创建版本快照，可回溯切换

**输出物**：CPG 因果图（节点 + 因果边）

### Phase 4：血肉阶段

**目的**：为每个骨架节点生成详细的 Story Beat（故事节拍）。

**流程**：

1. **选择人格**：从 10 个预设人格中选择参与生成的人格组合
2. **AI-Call-4 盲视变异**（温度 1.0，高发散度）：
   - 每个激活人格独立、并行生成同一节点的 Beat
   - 人格之间互不可见（盲视），确保方案多样性
   - 每个 Beat 包含：场景、角色、因果事件链、悬念钩子、创作理由
3. **选择确认**：
   - 卡片视图展示各人格方案
   - 可读视图 / JSON 编辑视图切换
   - 用户选择最佳方案并确认
4. **批量模式**：
   - 一键批量生成：逐节点生成，每个需人工确认
   - 全自动模式：自动选择第一个有效变体并确认，全部完成后自动进入扩写
5. **AI-Call-5 ITE 因果蒸馏**（温度 0.3）：
   - 评估每个事件对终局达成的因果贡献度 τᵢ
   - τᵢ < 0.05 标记为冗余，建议剔除
   - 检查相邻 Hauge 阶段间是否存在高冲击事件
6. **AI-Call-6 RAG 一致性审查**（温度 0.1）：
   - 新 Beat → Embedding → FAISS 检索 Top-K 相关文档
   - 只传相关片段给 AI 做矛盾检测（5 类矛盾：实体状态/规则违反/时间线/角色行为/因果断裂）

**输出物**：每个节点的已确认 Story Beat

### Phase 5：扩写阶段

**目的**：将 Story Beat 扩写为标准短剧剧本格式。

**流程**：

1. **AI-Call-7 剧本扩写**（温度 0.7）：
   - 通过因果图入边获取前情上下文（而非线性顺序）
   - 注入前序节点结尾片段（最后 500 字）作为衔接参考
   - 严格遵循剧本格式规范（集标题头/角色表/场景头/对白格式）
   - 字数控制：按每分钟 180 字估算
2. **编辑与保存**：可在编辑框内修改扩写结果
3. **批量全部重写**：重置所有已扩写状态后全部重新生成
4. **导出**：按节点编号排序导出完整剧本文本文件

**输出物**：标准格式剧本正文

### Phase 6：锁定阶段

**目的**：最终审阅和导出。

**流程**：统计表格展示全部节点状态，支持导出最终剧本。

## 第二部分：涉及理论

### 1. Michael Hauge 六阶段叙事框架

**来源**：好莱坞编剧顾问 Michael Hauge 的电影结构理论。

**六个阶段**：

| 阶段 | 名称 | 占比 | 叙事目标 |
|------|------|------|----------|
| 1 | 机会 (Opportunity) | 10-15% | 打破日常的契机，建立初始世界观 |
| 2 | 变点 (Change of Plans) | 10-15% | 主角被迫改变目标，踏上新道路 |
| 3 | 无路可退 (Point of No Return) | 15-20% | 全力投入，不可能回头 |
| 4 | 主攻/挫折 (Major Setback) | 20-25% | 最大障碍出现，一切似乎要失败 |
| 5 | 高潮 (Climax) | 15-20% | 终极对决，所有因果线汇聚 |
| 6 | 终局 (Aftermath) | 5-10% | 新秩序建立，角色弧线完成 |

**应用方式**：CPG 骨架生成时，将总集数按上述比例分配到六个阶段，确保宏观叙事弧线的完整性。同时要求每一集内部也具备「小起承转合」的微观弧线。

### 2. CPG 因果概率图 (Causal Probability Graph)

**原理**：将剧本结构建模为有向无环图（DAG），节点代表集/场景，边代表因果关系。

**因果类型**：
- **直接因果**：事件 A 直接导致事件 B
- **间接影响**：事件 A 间接促成事件 B 的条件
- **情感驱动**：角色情感变化驱动后续剧情走向
- **并行主题**：两条线并行推进，主题呼应
- **伏笔回收**：后续节点回收前面埋下的伏笔
- **冲突升级**：在前序基础上升级冲突强度

**约束规则**：
- 不允许「孤岛节点」（没有因果连接）
- 所有因果链最终指向终局条件
- 扩写时通过入边获取真正的前情来源

### 3. BVSR 盲视变异与选择保留

**来源**：NarrativeLoom 论文中的 Multi-Persona Ensemble 方法，借鉴进化论中的 Blind Variation and Selective Retention 机制。

**核心思想**：
- **盲视变异 (Blind Variation)**：多个 AI 人格独立、并行生成同一节点的方案，人格之间互不可见，确保方案多样性
- **选择保留 (Selective Retention)**：人类编剧从多个方案中选择最优方案

**10 个人格分类**：

| 类别 | 人格 | 专长 |
|------|------|------|
| 启动型 (Initiator) | 历史研究者 | 历史原型、权力更迭 |
| 启动型 | 反乌托邦远见者 | 压迫系统、个体对抗 |
| 启动型 | 科幻未来主义者 | 科学概念、技术伦理 |
| 推进型 (Developer) | 悬疑解谜者 | 信息差、顿悟时刻 |
| 推进型 | 政治策略家 | 权力博弈、两难选择 |
| 推进型 | 战争编年史家 | 军事冲突、战略叙事 |
| 氛围型 (Atmosphere) | 恐怖氛围营造者 | 心理恐怖、认知失调 |
| 氛围型 | 浪漫织梦者 | 情感纽带、情感两难 |
| 终结型 (Finisher) | 悲剧建筑师 | 必然悲剧、致命缺陷 |
| 终结型 | 救赎叙事者 | 内在转变、角色成长 |

### 4. ITE 个体处理效应 (Individual Treatment Effect)

**来源**：因果推断理论中的反事实分析方法。

**公式**：τᵢ = Ŷᵢ(1) - Ŷᵢ(0)

- Ŷᵢ(1)：保留事件 i 时，终局条件达成的合理性
- Ŷᵢ(0)：移除事件 i 后，终局条件仍能达成的合理性
- τᵢ 越接近 0，事件越冗余

**判定标准**：
- τᵢ < 0.05 → 冗余事件，建议剔除
- 相邻 Hauge 阶段间至少一个 τᵢ > 0.15（高冲击事件要求）

**实现方式**：LLM-as-Judge——让 Gemini 模拟因果推断，评估每个事件的贡献度。体现「台上没有一颗扣子是多余的」创作原则。

### 5. RAG 检索增强生成 (Retrieval-Augmented Generation)

**原理**：将世界观变量和已确认 Beat 向量化存入 FAISS 索引，新 Beat 审查时只检索 Top-K 相关文档传给 AI，而非全量上下文。

**矛盾检测维度**（5 类）：
1. 实体状态矛盾：已销毁/死亡的实体再次出现
2. 规则违反：新事件违反世界观规则
3. 时间线矛盾：事件时序冲突
4. 角色行为矛盾：行为与已确立的动机/性格不符
5. 因果断裂：事件缺乏因果前提

### 6. 苏格拉底诘问法 (Socratic Method)

**原理**：通过精准追问帮助用户锁定故事核心变量，而非直接告诉用户答案。问题之间有逻辑递进关系，后面的问题基于前面的假设。

### 7. 角色重要性分级体系

**三级分级**：
- **A级（核心）**：完整动机弧线，每集有戏份，情感变化线
- **B级（重要）**：有明确功能且反复出场，有限的情感变化
- **C级（工具）**：功能性出场，不做动机转变，不赋予内心戏

**创作纪律**：C级角色绝不做动机转变、不写内心独白，只通过行动体现功能。

### 8. 双层叙事结构

- **宏观弧线**：全剧遵循 Hauge 六阶段的大起承转合
- **微观弧线**：每一集内部也有自己的小起承转合（开头承接悬念→发展推进→转折高潮→结尾留钩子）

### 9. 题材预设风格体系

系统内置 7 种题材预设，每种预设包含骨架风格块、变异风格块、扩写风格块，从结构到细节全方位约束 AI 的创作方向：

犯罪/悬疑、爱情/情感、惊悚/恐怖、复仇/逆袭、奇幻/玄幻、都市/职场、喜剧/荒诞、自定义

## 第三部分：涉及技术

### 1. AI 模型与调用

| 技术 | 说明 |
|------|------|
| **Gemini 2.5 Flash** | Google 大语言模型，通过 VertexAI 端点调用 |
| **google-genai SDK** | REST 模式调用（兼容 ARM64 Windows，不依赖 gRPC） |
| **Gemini text-embedding-004** | 768 维文本向量模型，用于 RAG 向量检索 |
| **服务账号认证** | Google Cloud Service Account JSON 密钥 |
| **本地代理** | 通过 HTTP 代理访问 VertexAI API |

**AI 调用编号与温度策略**：

| 编号 | 调用 | 温度 | 说明 |
|------|------|------|------|
| AI-Call-1 | 苏格拉底盘问 | 0.4 | 精准追问，低发散 |
| AI-Call-1.5 | 角色生成 | 0.5 | 适度创意 |
| AI-Call-2 | 世界观提炼 | 0.2 | 信息提炼，极低发散 |
| AI-Call-3 | CPG 骨架生成 | 0.6 | 需要创意但需结构 |
| AI-Call-4 | 盲视变异 | 1.0 | BVSR 核心，高发散度 |
| AI-Call-5 | ITE 因果蒸馏 | 0.3 | 严格逻辑分析 |
| AI-Call-6 | RAG 一致性审查 | 0.1 | 最严格判断 |
| AI-Call-7 | 剧本扩写 | 0.7 | 创意+格式控制 |

### 2. 并行调用与 QPS 控制

- **asyncio + 信号量**：最大并发 3 个 API 调用
- **随机间隔**：每次调用前等待 1-5 秒，避免触发 API 限流
- **线程池执行器**：`loop.run_in_executor` 将同步 API 包装为异步调用

### 3. GUI 框架

| 技术 | 说明 |
|------|------|
| **PySide6 (Qt6)** | 跨平台桌面 GUI 框架 |
| **QThread** | 后台工作线程，所有 AI 调用异步执行，UI 不阻塞 |
| **Signal/Slot** | Qt 信号槽机制，Worker 线程与 UI 线程安全通信 |
| **QStackedWidget** | 多阶段页面切换 |
| **QScrollArea** | 防止窗口过小时内容挤压 |
| **QGraphicsView/Scene** | 力导向人物关系图的底层渲染引擎 |

### 4. 向量检索

| 技术 | 说明 |
|------|------|
| **FAISS (IndexFlatIP)** | Facebook 开源的高性能向量检索库 |
| **内积 + L2 归一化** | 等价于余弦相似度 |
| **Top-K 检索** | 只将最相关的 8 条文档传给 AI 审查 |
| **懒初始化** | 首次使用时才创建索引 |

### 5. 数据持久化

| 技术 | 说明 |
|------|------|
| **ProjectData dataclass** | 项目全状态的可序列化数据类 |
| **.story.json** | 项目存档格式，JSON 序列化 |
| **config/*.json** | 人格配置和题材预设的持久化文件 |
| **操作历史栈** | 支持操作回退的历史记录 |
| **节点版本快照** | 每次修改自动创建版本，支持回溯 |

### 6. JSON 处理与容错

- **response_mime_type**：强制 AI 返回 `application/json` 格式
- **截断 JSON 修复**：`_repair_truncated_json()` 自动闭合未闭合的字符串/括号
- **多模式解析**：支持从代码块提取、宽松匹配 `{...}` / `[...]`、截断修复

### 7. Prompt 工程

| 技术 | 说明 |
|------|------|
| **System/User Prompt 分离** | System 定义角色和规则，User 提供具体任务 |
| **str.replace 模板** | 避免 `.format()` 与 JSON 花括号冲突 |
| **风格块注入** | 根据题材和风格动态拼接 Prompt 片段 |
| **人格身份块** | 每个人格独立的 System Prompt 身份层 |
| **因果上下文注入** | 入边/出边关系描述注入 Prompt |

### 8. 架构模式

| 模式 | 说明 |
|------|------|
| **MVC 分层** | models（数据）/ services（业务）/ ui（界面）三层分离 |
| **全局单例** | ai_service、persona_engine、ite_calculator、rag_controller、genre_manager |
| **Worker 线程模式** | 每种 AI 调用对应独立的 QThread Worker 类 |
| **信号驱动** | Phase 完成 → 信号 → 主窗口切换下一阶段 |
| **阶段状态机** | genesis → characters → skeleton → flesh → expansion → locked |

### 9. 关键依赖

| 依赖 | 用途 |
|------|------|
| `google-genai` | Gemini API 调用（REST 模式） |
| `google-auth` | Google Cloud 服务账号认证 |
| `PySide6` | Qt6 GUI 框架 |
| `faiss-cpu` | 向量相似度检索 |
| `numpy` | 向量计算 |

### 10. 两遍生成法（大集数优化）

当总集数 > 30 时，单次生成可能超出 token 限制，采用两遍生成法：

1. **Pass 1 — 全局大纲**：一次调用生成所有集的紧凑大纲（每集仅一行摘要）
2. **Pass 2 — 逐阶段展开**：以完整大纲为上下文，逐阶段展开为详细节点
