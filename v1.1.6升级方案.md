## 一、问题诊断（一句话总结）

> **AI 剧本不是写得不好，是写在了错误的"颗粒度"上。**
> 人工剧本颗粒度 = 因果事件；AI 剧本颗粒度 = 动作切片。
> 人工剧本《结婚》Ep1 装下「重生顿悟+仇人画像+主线确立」≈5 个因果事件；AI 剧本《巫祝血祭》Ep7-Ep13 共 7 集只装下 1 个事件（打古殿邪灵），密度差距 35 倍以上。

### 1.1 AI vs 人工 — 量化差距表

| 维度 | 人工剧本 | AI 剧本 | 差距 |
|---|---|---|---|
| 首集因果事件数 | 5 个 | 1 个 | 5× |
| 主线明示集 | Ep1 | Ep15 / Ep26 | 26× |
| 单一物理场景最大跨集 | ≤2 集 | 7 集（古殿） | 3.5× |
| CP 互动密度 | 每集 ≥1 处 | 全剧 0 处 | ∞ |
| 人物变化 | Ep1 即立标签 | Ep1=Ep30=惊恐 | 0 弧线 |
| 钩子类型分布 | 信息/反转/动机/危机混用 | 20+/30 集"危机定格" | 单调 |

### 1.2 软件层面 6 大根因

| 编号 | 根因 | 直接症状 | 修复优先级 |
|---|---|---|---|
| R1 | CPG 颗粒度强行绑死「1 集 = 1 节点」 | 节奏慢 / 密度低 | P0 |
| R2 | ITE 是评分器不是裁剪器（无反馈闭环） | 拖沓（7 集打 1 怪） | P0 |
| R3 | 同场景停留无上限约束 | 困古殿 7 集 | P0 |
| R4 | 钩子库无类型配比强制 | 钩子全是"危机定格" | P1 |
| R5 | 角色卡缺 signature_traits / arc 字段 | 人物扁平 | P2 |
| R6 | 完全没有 CP 互动 prompt 钩子 | 平淡叙事 | P2 |
| **R7** | **CP 互动注入位置不明，会污染骨架/扩写** | **若错误注入会破坏 ITE 与节奏** | **P3.5（铁律）** |

## 二、流程评估（不需要大改）

### 2.1 流程结构保留

6 阶段方向完全正确，**保留**：

```
创世 → 人物 → 骨架 → 血肉 → 扩写 → 锁定
```

### 2.2 流程闭环新增 1 处（关键）

**骨架阶段新增 ITE 自动裁剪闭环**：

```
旧：骨架生成 → 用户确认 → 进入血肉
新：骨架生成 → ITE 即时评分 → 自动合并冗余节点（τ<0.05 连续≥3 个） → 用户确认 → 进入血肉
```

ITE 从「血肉阶段事后审查」前移为「骨架阶段事中裁剪」。

### 2.3 数据颗粒度调整 1 处

```
旧：1 集 = 1 个 CPG 节点（节点 ID = Ep1, Ep2...）
新：1 集 = 2-4 个事件级 CPG 节点（节点 ID = Ep1-A, Ep1-B, Ep2-A...）
   集容器（Episode Container）汇总该集所有节点
```

### 2.4 不动的部分

- ✅ Phase 1 创世 + 苏格拉底盘问
- ✅ Phase 2 人物分级（A/B/C）+ 关系网
- ✅ Phase 4 血肉 BVSR 多人格盲视变异
- ✅ Phase 5 扩写（因果图驱动上下文）
- ✅ Phase 6 锁定 / 导出
- ✅ RAG 一致性审查
- ✅ Hauge 六阶段宏观弧线
- ✅ 题材预设（仅重写内部 prompt 内容，结构不动）
- ✅ 爽感公式系统、钩子公式系统、写作铁律（v1.1.2 引入的内容保留）

#### P2-3 CP 互动配额（重写版 — v1.1.6.2 关键修正）

> ⚠️ **【架构纠正】** CP 互动模板库 **只能在血肉阶段（Phase 4 Beat 生成）调用**，骨架阶段与扩写阶段必须显式禁止。
>
> **理由**：
> - **骨架阶段**：CPG 节点是因果事件颗粒，CP 互动是情绪细节。混入会污染因果链，破坏 ITE 计算（τ 值失真）。
> - **血肉阶段**：唯一正确战场。Beat 是「事件→动作」的转译节点，CP 互动天然嵌入而不突兀。
> - **扩写阶段**：只做镜头化润色，二次注入 CP 互动会拖慢节奏、灌水、破坏单集闭环、与 ITE 压缩冲突。

##### 改动内容（按文件）

**A. `services/cp_interaction_engine.py`（新建）**
```python
class CPInteractionEngine:
    """CP 互动模板抽样引擎。仅在血肉阶段（Phase 4）被 worker 调用。"""

    STAGE_WHITELIST = {"flesh"}  # 仅血肉阶段允许调用

    def __init__(self, template_path="config/cp_interaction_templates.json"):
        self._load_templates(template_path)

    def sample(self, episode_node, project_data, stage="flesh"):
        # 阶段守卫：非血肉阶段调用 → 抛出 RuntimeError
        if stage not in self.STAGE_WHITELIST:
            raise RuntimeError(
                f"CP 互动模板库禁止在 {stage} 阶段调用。"
                f"仅允许在 {self.STAGE_WHITELIST} 中使用。"
            )
        # 1. 题材映射 → 候选库
        # 2. Hauge 阶段过滤
        # 3. 钩子配比约束（同类型不连续 3 集）
        # 4. 占位符注入 {role_a}/{role_b}/{scene}/{prop}/{villain}/{core_setting}/{plot_node}
        # 5. 兜底：含 {villain} 的模板若本集无反派则跳过
        return rendered_content
```

### P2-3.5 — 血肉阶段主 Prompt 直改清单（v1.1.6.3 关键新增）

> ⚠️ **这是本次升级的"心脏"。** 所有针对"情节密度、剧情转折、CP 互动、人物微变化"的增强，**必须直接改写 `SYSTEM_PROMPT_VARIATION_FRAME` 主体**，而不是附加段落或独立 prompt。AI 对附加段的遵从度远低于主 prompt 主体规则。

#### 核心目标
让血肉阶段产出的每一个 Beat 都满足：
1. **高情节密度**：1 集 ≥ 3 个独立因果事件，无水分
2. **强转折/钩子**：每个 Beat 必须有 ≥1 处反转 / 信息突破 / 立场翻转
3. **CP 拉扯落地**：若 has_cp_main_line=True，必须嵌入 1 处来自 cp_interaction_engine 的 CP 互动（占比 ≤ 30%）
4. **人物微变化外化**：A 级角色每集必须有 1 处 character_micro_change（动作/台词外化，禁止内心独白）

#### `SYSTEM_PROMPT_VARIATION_FRAME` 主 prompt 改写点（10 个章节）

| 序号 | 现状 | 改成什么 |
|---|---|---|
| § 创作任务 | "为指定的 CPG 节点生成一个详细的 Story Beat" | 加："**每个 Beat 必须包含 ≥3 个独立因果事件，每个事件都能独立成戏，禁止动作切片堆叠**" |
| § 角色重要性等级 | 已有 A/B/C 分级 | 加："**A 级角色在本集必须有 1 处 character_micro_change（认知更新/情感升级/立场转移/关系深化），通过具体动作或台词外化，禁止内心独白**" |
| § 创作约束（不可违反）#4 事件数量要求 | "Hauge 阶段 1-2：3 个；阶段 3：3-4 个；阶段 4-5：4-5 个；阶段 6：3 个" | **改为**："Hauge 阶段 1-2：≥3 个**独立因果事件**；阶段 3：≥4 个；阶段 4-5：≥5 个；阶段 6：≥3 个。**每个事件 τ_estimate ≥ 0.1，τ<0.05 的事件视为水分必须剔除**" |
| § 创作约束 #5 因果影响 | "每个事件必须标注'因果影响'" | 加："**每集必须有 ≥1 处剧情转折（反转/信息突破/立场翻转/秘密暴露），通过 events[i].twist_type 字段标记**" |

#### 阶段隔离的实现保障

1. **代码层**：`CPInteractionEngine.STAGE_WHITELIST = {"flesh"}`，非白名单阶段调用直接抛 RuntimeError
2. **Prompt 层**：三段铁律分别注入到 `SYSTEM_PROMPT_CPG_SKELETON`（骨架）/ `SYSTEM_PROMPT_VARIATION_FRAME` + `USER_PROMPT_VARIATION`（血肉，BVSR 主 prompt）/ `SYSTEM_PROMPT_EXPANSION`（扩写）—— v1.1.6.3 已修正常量名
3. **Worker 层**：仅 `VariationWorker`（血肉变异 worker，源码现状）持有 `cp_engine` 引用，骨架/扩写 Worker 不创建该实例
4. **数据层**：`cp_interaction_templates.json` 的 meta_info 显式声明 `stage_whitelist` 和 `stage_blacklist`
5. **UI 层**：扩写阶段如检测到生成结果含未在 Beat 中出现的 CP 互动关键词（撩拨/吃醋/双向奔赴等核心词），给出警告

## 五、验收标准（通用化 — v1.1.6.3 重写）

### 5.2 验收方式

1. **单项验收**：每条阈值独立校验，未达标项需在升级日志中说明原因或回滚相应改动。
2. **整体回归**：用任一既有剧本的 Sparkle 重新跑全流程，对比 v1.1.5 与 v1.1.6 同 Sparkle 输出。
3. **样本剧本**：建议至少覆盖以下 3 类（任选已有 Sparkle 即可，不限定具体作品）：
   - **奇幻/惊悚类**（验证 fantasy_thriller CP 库 + 高密度战斗事件不再被拆切片）
   - **重生/逆袭类**（验证 rebirth_revenge CP 库 + 主线节奏锁）
   - **现言/甜宠类**（验证 modern_sweet CP 库 + CP 互动配额）
4. **集数无关**：阈值表中所有"位置类"指标均使用**百分比**（C1-C3），保证对 12 集 / 30 集 / 80 集等任意长度剧本都适用。
5. **可选启用项**：CP 互动相关验收（F 组）仅在 `has_cp_main_line=True` 时启用；不启用时 F 组全部跳过。

### 5.3 自动化验收脚本（v1.1.7 计划）

下一版本将提供 `tools/acceptance_check.py`，自动化执行 A-H 所有维度的扫描与统计，输出 PDF 报告。本版（v1.1.6）暂以人工抽样 + 单元测试覆盖为主。

## 七、风险与回滚

| 风险 | 应对 |
|---|---|
| CPG 节点结构变更可能导致旧项目存档不兼容 | 加版本字段 `cpg_schema_version`；旧项目自动升级（每集 1 节点 → 转换为 1 集 1 节点的容器） |
| ITE 自动裁剪误删用户想保留的节点 | 默认仅"建议"不"强制"；用户可逐条接受/拒绝；保留版本快照 |
| 钩子配比强制可能与用户自定义钩子公式冲突 | 用户自定义钩子公式优先级 > 配比强制 |
| 短剧 prompt 改动影响传统影视风格 | 所有改动用 `if drama_style == "short_drama"` 分支隔离 |
| CP 引擎在错误阶段被误调用 | STAGE_WHITELIST 守卫 + RuntimeError + 单元测试覆盖 |
| 扩写阶段 AI 偷偷新增 CP 互动 | 扩写后扫描关键词列表，命中则警告并允许用户手动剔除 |
| 骨架阶段 AI 把 CP 情绪写入 event_summary | 骨架 prompt 显式禁止 + 生成后用关键词扫描器校验 |
| **【v1.1.6.3】主 prompt 改写后 AI 不返回新增字段（density_score/twist_type 等）** | **schema 校验 + 缺字段时强制重试一次 + 仍缺则 worker 自动补 0 并标黄警告** |
| **【v1.1.6.3】主 prompt 过长导致 token 超限** | **超过 8K token 时启用精简版 prompt（合并相邻章节，仅保留硬约束）** |
| **【v1.1.6.3】Beat schema 升级使旧项目存档无法加载** | **加版本字段 `beat_schema_version`；旧 Beat 自动补全缺失字段为 None/0** |
